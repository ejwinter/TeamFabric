# Skill: Retro Report

## Purpose

Synthesize all member inputs from a retrospective folder into a structured team summary. Identifies themes across responses, consolidates action items, and produces `summary.md` in the retro folder. Invoked by `/retro close`.

## Invocation

Invoked internally by `/retro close` after the user confirms they want to close the retro. Receives the retro folder path as context. Runs silently; the result is previewed to the user before being written.

## Steps

### 1. Load the Retro Definition

Read `<retro-folder>/retro.md`. Extract:

- Period (start and end dates)
- Iteration name (if set)
- Questions (custom list or `default`)
- Participant table (who submitted, who is pending)
- Created by (for attribution in the summary header)

If questions are `default`, use the framework default set:

1. What went well this period?
2. What didn't go well or felt harder than it should have?
3. What should we change or improve going forward?
4. Any action items?

If the team has defined a custom default in their CLAUDE.md constitution (`### Retrospective` / `Default questions:`), use those instead of the framework default.

### 2. Load All Member Inputs

For each submitted member (those with `Submitted` in the participant table), read `<retro-folder>/<member-slug>.md`.

Record the member name and their answers for each question section. Also collect any action items from their `## Action Items` section.

Do not load pending members — note them as having no input.

### 3. Synthesize Themes by Question

For each question, scan all member answers and identify recurring patterns:

- **Cluster**: group responses that describe the same underlying theme, even if worded differently.
- **Name each cluster**: give the cluster a short, concrete label (e.g. "CI pipeline reliability", "unclear handoff from triage", "cross-team dependencies").
- **Count**: note how many members mentioned each theme.
- **Preserve specifics**: include concrete details (item names, incident descriptions) rather than averaging into vague summaries.
- **Flag recurring patterns** for the "What Didn't Go Well" question — if the same issue came up last retro as well (check prior `summary.md` if available), mark it as recurring.

Minority responses (mentioned by only one member) that are distinct and substantive should appear as standalone items, not be dropped.

### 4. Consolidate Action Items

Collect all action items from member inputs. For each:

- **De-duplicate**: merge items that are substantively the same.
- **Confirm or assign ownership**: use the owner named in the member's input if present. If unowned, suggest the most appropriate member based on role/domain and mark it as a suggestion (`*suggested*`).
- **Flag vague items**: action items that are too vague to track (e.g. "improve communication") should be flagged with a note that they need refinement before being tracked.

### 5. Produce the Draft

Generate the draft content for `output/retro-<retro-id>.md`. The draft includes synthesized themes and action items, plus addressal placeholder sections for the lead to fill in before finalizing the close.

```markdown
# Retrospective Draft — [retro-id] — YYYY-MM-DD

> Period: YYYY-MM-DD to YYYY-MM-DD[  |  Iteration: [Name] if applicable]
> Created by: [Name]
> Participants ([N]/[total]): [names who submitted]
> No input: [names who were pending, or "All members contributed"]

<!--

  You can edit this file directly or ask the agent to help you work through each point.
-->

---

[2–4 bullets with specifics]
```

Return this content to `/retro close`. Do not write the file — `/retro close` writes it to `output/retro-<retro-id>.md`.

## Tone and Length

- Theme sections: preserve specifics over generic paraphrasing. 2–4 bullets per theme. Don't inflate thin coverage.
- Recurring patterns should be labeled prominently — a theme that keeps coming back is more urgent than a new one.
- Action items must be specific and owned (or explicitly flagged as needing ownership). Never drop an action item silently.
- Participation header: factual. No editorial commentary on who didn't contribute.
- If fewer than half of expected members contributed, note this prominently at the top:
  > "⚠ Low participation: [N] of [total] members contributed. Themes may not reflect the full team's perspective."

## Notes

- Read-only. Does not write any files — returns synthesized content to the calling command.
- If a prior retro summary exists anywhere in `team/retros/`, read the most recent one to detect recurring themes. If none exists, skip recurring detection.
